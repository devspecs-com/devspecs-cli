# PRD: DevSpecs v0 CLI

## 1. Product Summary

DevSpecs is a local-first CLI for indexing, identifying, and referencing software planning/specification artifacts across a developer’s projects.

v0 focuses on local capture and resolution only.

The CLI should scan a repository for existing planning/spec artifacts such as OpenSpec changes, ADRs, markdown plans, and generic spec files, store normalized metadata in a local SQLite index, assign stable DevSpecs IDs, and provide simple commands for listing, showing, resolving, and exporting agent-ready context.

DevSpecs v0 is **not** a full cloud platform, search engine, governance workflow, or provenance/intelligence system. It should create the foundation for later cloud sync, GitHub/Linear linkage, MCP integration, and Kleio ingestion.

## 2. Core Goal

Give every local software spec, plan, ADR, or proposal a stable local ID and make it easy for humans and agents to resolve that ID back into useful context.

Primary product promise:

> Index your plans and specs locally. Give them stable IDs. Reference them from agents, PRs, issues, and future workflows.

## 3. Non-Goals for v0

Do not implement:

- Cloud sync
- User accounts/auth
- Team workspaces
- GitHub OAuth
- GitHub App
- Linear integration
- CI/CD gating
- Approval workflows
- Merge requests/change reviews
- Semantic search
- Embeddings
- Automatic agent session mining
- Incident tracing
- Implementation drift detection
- Deep provenance graph reconstruction
- Kleio integration beyond future-compatible schema design
- Kalo integration
- Complex lifecycle governance
- Multi-user collaboration

v0 should remain a focused local CLI with a SQLite index.

## 4. Product Boundary

DevSpecs answers:

- What specs/plans/ADRs exist in this repo?
- What stable local ID identifies this artifact?
- Where did this artifact come from?
- What is the latest observed revision?
- What context should I pass to an AI agent for this artifact?

DevSpecs does **not** answer:

- Why did implementation diverge?
- What happened historically around this plan?
- What caused an incident?
- Which agent made which decision?
- How did the full engineering intent graph evolve?

Those are future Kleio-style responsibilities.

## 5. Target User

Primary user:

- Senior developer or AI-assisted developer using tools like Cursor, Claude Code, OpenSpec, GitHub Issues, ADRs, and markdown planning files.
- Works across one or more repos.
- Wants lightweight structure without adopting a heavy planning/governance platform.
- Wants to preserve useful AI-generated plans/specs beyond transient chat sessions.

Secondary future users:

- Small engineering teams using mixed AI tools.
- Teams adopting spec-driven development.
- Developers who want PRs/issues to reference durable specs.

## 6. Key Concepts

### 6.1 Artifact

A logical planning/specification object.

Examples:

- OpenSpec proposal
- OpenSpec spec delta
- ADR
- Markdown implementation plan
- Cursor-generated plan
- Claude-generated plan
- Requirements document
- Technical design document
- Task list

An artifact has a durable DevSpecs ID.

### 6.2 Revision

A specific observed content snapshot of an artifact.

Each time the artifact content changes, DevSpecs should be able to detect a new revision using content hashing.

v0 does not need advanced diff UI, but it should store enough data to support revision history later.

### 6.3 Source

Where the artifact came from.

Examples:

- Local file path
- OpenSpec change directory
- ADR markdown file
- Generic markdown file
- Future: GitHub issue, Linear issue, Cursor export, Claude export

### 6.4 Adapter

A parser/extractor for known artifact formats.

v0 adapters:

- OpenSpec adapter
- ADR markdown adapter
- Generic markdown plan/spec adapter

Optional if easy:

- Cursor/Claude exported markdown adapter, treated mostly as generic markdown

### 6.5 DevSpecs ID

A stable local ID assigned to each logical artifact.

Example:

```txt
ds_01JY8ZP9Q8F7B6M4K2R3T1V0WX
````

Use ULID or another sortable unique ID format.

The ID should remain stable across rescans as long as DevSpecs can match the artifact to its source identity.

## 7. v0 User Experience

### 7.1 Install

Implementation can assume local development for now, but CLI design should support later packaging.

Preferred binary name:

```bash
ds
```

Product/package name:

```txt
devspecs
```

### 7.2 Initialize

Command:

```bash
ds init
```

Behavior:

* Creates global DevSpecs directory if missing:

```txt
~/.devspecs/
~/.devspecs/devspecs.db
```

* Creates optional repo-local config in current repo:

```txt
.devspecs/config.yaml
```

* If no git repo is detected, still allow init but mark repo metadata as unknown.
* If already initialized, print a friendly message and do not overwrite existing config unless `--force` is provided.

Example output:

```txt
Initialized DevSpecs.

Global index:
  ~/.devspecs/devspecs.db

Repo config:
  .devspecs/config.yaml

Next:
  ds scan
```

### 7.3 Scan Repository

Command:

```bash
ds scan
```

Behavior:

* Scans current repository for supported artifact sources.
* Extracts artifact metadata.
* Stores artifacts, sources, and revisions in SQLite.
* Reuses existing artifact IDs when rescanning the same source.
* Detects changed content by hash and creates a new revision.
* Prints a concise summary.

Example output:

```txt
Scanned repository: acme/backend

Found:
  4 OpenSpec changes
  3 ADRs
  5 markdown plans

Indexed:
  2 new artifacts
  1 updated artifact
  9 unchanged artifacts

Run:
  ds list
```

Flags:

```bash
ds scan --path .
ds scan --verbose
ds scan --json
```

### 7.4 List Artifacts

Command:

```bash
ds list
```

Behavior:

* Lists artifacts known for the current repo.
* Default ordering: most recently observed/updated first.
* Should show ID, kind, title, status if known, source type, and source path.

Example:

```txt
ID            KIND       STATUS      TITLE
ds_01JY8...   openspec   proposed    Add work item canonical vocabulary
ds_01JY7...   adr        accepted    Use SQLite for local index
ds_01JY6...   plan       draft       Refactor artifact ingestion pipeline
```

Flags:

```bash
ds list --kind adr
ds list --kind openspec
ds list --status draft
ds list --source openspec
ds list --json
```

### 7.5 Show Artifact

Command:

```bash
ds show <id>
```

Behavior:

* Shows normalized metadata and content preview/full content.
* Default output should be human-readable.
* Include source path and current revision hash.
* If the ID is ambiguous or unknown, return a useful error.

Example:

```txt
DevSpec: ds_01JY8ZP9Q8F7B6M4K2R3T1V0WX

Title:
  Add work item canonical vocabulary

Kind:
  openspec_change

Status:
  proposed

Source:
  openspec/changes/add-work-item-canonical-vocabulary/proposal.md

Current revision:
  sha256:abc123...

Extracted:
  Acceptance criteria: 4
  Tasks: 7
  Risks: 2

Content:
  ...
```

Flags:

```bash
ds show <id> --json
ds show <id> --content
ds show <id> --no-content
```

### 7.6 Find Artifacts

Command:

```bash
ds find <query>
```

Behavior:

* Basic local search over title, source path, kind, status, and optionally body text.
* Use SQLite FTS if straightforward.
* This is not semantic search.
* Do not market this as the core feature; it is lookup/discovery.

Example:

```bash
ds find auth
```

Example output:

```txt
ID            KIND       TITLE
ds_01JY9...   plan       Add Google OAuth login
ds_01JY3...   adr        Use Auth.js for authentication
```

Flags:

```bash
ds find "auth" --json
ds find "auth" --kind adr
```

### 7.7 Resolve Artifact

Command:

```bash
ds resolve <id>
```

Behavior:

* Resolves an ID to its canonical local source and metadata.
* Intended for scripts/agents.
* Should be stable and concise.

Example:

```bash
ds resolve ds_01JY8
```

Output:

```txt
ds_01JY8ZP9Q8F7B6M4K2R3T1V0WX
openspec/changes/add-work-item-canonical-vocabulary/proposal.md
```

JSON:

```bash
ds resolve ds_01JY8 --json
```

```json
{
  "id": "ds_01JY8ZP9Q8F7B6M4K2R3T1V0WX",
  "kind": "openspec_change",
  "title": "Add work item canonical vocabulary",
  "source_path": "openspec/changes/add-work-item-canonical-vocabulary/proposal.md",
  "current_revision_hash": "sha256:abc123"
}
```

### 7.8 Agent Context Export

Command:

```bash
ds context <id>
```

Behavior:

* Outputs an agent-ready context bundle for the artifact.
* The output should be plain markdown by default.
* It should include title, kind, status, source, summary if extracted, acceptance criteria/tasks if extracted, related links if any, and the artifact body.

Example:

```bash
ds context ds_01JY8
```

Output shape:

```md
# DevSpecs Context: Add work item canonical vocabulary

DevSpec ID: ds_01JY8ZP9Q8F7B6M4K2R3T1V0WX
Kind: openspec_change
Status: proposed
Source: openspec/changes/add-work-item-canonical-vocabulary/proposal.md

## Instructions for Agent

Use this artifact as the source of truth for the requested implementation or review.

Preserve the acceptance criteria.
Do not silently change scope.
If implementation diverges from this artifact, explicitly record the deviation.

## Extracted Acceptance Criteria

- ...
- ...

## Extracted Tasks

- ...
- ...

## Source Content

...
```

Flags:

```bash
ds context <id> --json
ds context <id> --copy
```

`--copy` is optional. If implemented, copy output to clipboard.

### 7.9 Capture Explicit File

Command:

```bash
ds capture <path>
```

Behavior:

* Adds a specific file as an artifact even if it is not discovered by scan.
* Useful for one-off AI-generated plans.
* Detect kind using heuristics unless `--kind` is provided.

Example:

```bash
ds capture ./tmp/claude-plan.md --kind plan
```

Output:

```txt
Captured artifact:
  ds_01JY9Q...
  ./tmp/claude-plan.md
```

Flags:

```bash
ds capture <path> --kind plan
ds capture <path> --title "Refactor ingestion pipeline"
ds capture <path> --status draft
ds capture <path> --json
```

### 7.10 Basic Status Management

Command:

```bash
ds status <id> <status>
```

Supported statuses:

```txt
draft
proposed
approved
implementing
implemented
superseded
rejected
unknown
```

Behavior:

* Updates local artifact status.
* Records updated timestamp.
* Does not create formal approval workflow.

Example:

```bash
ds status ds_01JY8 approved
```

Output:

```txt
Updated ds_01JY8 status: proposed -> approved
```

### 7.11 Basic Link Management

Command:

```bash
ds link <id> <target>
```

Behavior:

* Adds a link from artifact to external/local target.
* Target can initially be a URL or a typed string.

Examples:

```bash
ds link ds_01JY8 https://github.com/acme/backend/pull/42
ds link ds_01JY8 github://acme/backend/pull/42
ds link ds_01JY8 linear://ACME-123
```

Optional flags:

```bash
ds link <id> <target> --type implements
ds link <id> <target> --type related
ds link <id> <target> --type supersedes
```

Supported link types:

```txt
related
implements
implemented_by
supersedes
superseded_by
blocks
blocked_by
references
referenced_by
```

v0 does not need validation of external targets.

## 8. Adapter Requirements

### 8.1 OpenSpec Adapter

Detect common OpenSpec structures:

```txt
openspec/
  specs/
  changes/
```

For `openspec/changes/<change-id>/proposal.md`:

* kind: `openspec_change`
* title: first markdown H1 if present, else change directory name
* status: infer from folder/content if possible, else `unknown` or `proposed`
* source path: proposal.md
* related files:

  * design.md
  * tasks.md
  * specs/**/*.md inside the change directory

For v0, it is acceptable to treat `proposal.md` as the primary source and include related files in extracted metadata if easy.

Extract tasks from `tasks.md` if present.

Task extraction heuristic:

* Markdown checklist items:

  * `- [ ] task`
  * `- [x] task`

Acceptance criteria extraction heuristic:

* Markdown section headings containing:

  * Acceptance Criteria
  * Requirements
  * Scenarios
  * Success Criteria

### 8.2 ADR Adapter

Detect files commonly matching:

```txt
docs/adr/*.md
docs/adrs/*.md
adr/*.md
adrs/*.md
architecture/decisions/*.md
```

Heuristics:

* kind: `adr`
* title: first H1, else filename
* status:

  * detect lines like `Status: Accepted`
  * detect headings/metadata containing status
  * else `unknown`
* source path: markdown file

Do not require a single ADR format.

### 8.3 Generic Markdown Plan/Spec Adapter

Detect likely plan/spec files in common locations:

```txt
docs/specs/**/*.md
specs/**/*.md
plans/**/*.md
docs/plans/**/*.md
.requirements/**/*.md
```

Also support direct `ds capture`.

Heuristics:

* kind:

  * filename/path contains `plan` → `plan`
  * filename/path contains `spec` → `spec`
  * filename/path contains `requirements` → `requirements`
  * otherwise `markdown_artifact`
* title: first H1, else filename
* status: infer from frontmatter if available, else `unknown`

Support YAML frontmatter if easy:

```yaml
---
title: Add SSO login
kind: plan
status: draft
---
```

Frontmatter should override heuristics.

## 9. Data Storage

### 9.1 Location

Global directory:

```txt
~/.devspecs/
```

SQLite DB:

```txt
~/.devspecs/devspecs.db
```

Repo config:

```txt
.devspecs/config.yaml
```

### 9.2 Suggested SQLite Tables

Minimum:

```sql
workspaces
repos
artifacts
artifact_revisions
sources
links
```

Optional:

```sql
tags
observations
scan_runs
```

### 9.3 Suggested Schema

#### repos

```sql
CREATE TABLE repos (
  id TEXT PRIMARY KEY,
  root_path TEXT NOT NULL,
  git_remote_url TEXT,
  git_current_branch TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
```

#### artifacts

```sql
CREATE TABLE artifacts (
  id TEXT PRIMARY KEY,
  repo_id TEXT,
  kind TEXT NOT NULL,
  title TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'unknown',
  canonical_source_id TEXT,
  current_revision_id TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  last_observed_at TEXT NOT NULL,
  FOREIGN KEY(repo_id) REFERENCES repos(id)
);
```

#### artifact_revisions

```sql
CREATE TABLE artifact_revisions (
  id TEXT PRIMARY KEY,
  artifact_id TEXT NOT NULL,
  content_hash TEXT NOT NULL,
  body TEXT NOT NULL,
  extracted_json TEXT,
  observed_at TEXT NOT NULL,
  git_commit TEXT,
  FOREIGN KEY(artifact_id) REFERENCES artifacts(id)
);
```

#### sources

```sql
CREATE TABLE sources (
  id TEXT PRIMARY KEY,
  artifact_id TEXT NOT NULL,
  repo_id TEXT,
  source_type TEXT NOT NULL,
  path TEXT,
  url TEXT,
  source_identity TEXT NOT NULL,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  FOREIGN KEY(artifact_id) REFERENCES artifacts(id),
  FOREIGN KEY(repo_id) REFERENCES repos(id)
);
```

#### links

```sql
CREATE TABLE links (
  id TEXT PRIMARY KEY,
  artifact_id TEXT NOT NULL,
  link_type TEXT NOT NULL,
  target TEXT NOT NULL,
  created_at TEXT NOT NULL,
  FOREIGN KEY(artifact_id) REFERENCES artifacts(id)
);
```

### 9.4 Identity Matching

On scan, DevSpecs should avoid creating duplicate artifacts for the same source.

Use `source_identity` as the primary matching key.

For local files:

```txt
repo_root + relative_path + adapter_type
```

For OpenSpec changes:

```txt
repo_root + openspec_change_id + adapter_type
```

This can evolve later.

### 9.5 Content Hashing

Use SHA-256 of normalized file content.

For multi-file artifacts such as OpenSpec changes, either:

* v0 simple approach: hash primary file only
* better approach: hash concatenated normalized content of primary + related files

Either is acceptable, but document the choice.

## 10. Repo Config

Create `.devspecs/config.yaml`.

Initial shape:

```yaml
version: 1

sources:
  - type: openspec
    path: openspec
  - type: adr
    paths:
      - docs/adr
      - docs/adrs
      - adr
      - adrs
  - type: markdown
    paths:
      - specs
      - docs/specs
      - plans
      - docs/plans
```

Allow users to edit this manually.

v0 does not need a config-editing command, but should read this file if present.

If no config exists, use sensible defaults.


Commands:

* `ds scan --json`

## 12. Error Handling

Principles:

* Fail clearly.
* Never corrupt the local DB.
* Do not crash on malformed markdown/frontmatter.
* Do not require Git.
* Give actionable next steps.

Examples:

Unknown ID:

```txt
Could not find artifact: ds_abc

Try:
  ds list
  ds find <query>
```

No index initialized:

```txt
DevSpecs is not initialized.

Run:
  ds init
```

No artifacts found:

```txt
No specs or plans found.

Try:
  ds capture <path>
  or edit .devspecs/config.yaml to add source paths.
```

Malformed config:

```txt
Invalid .devspecs/config.yaml: <error>

Fix the config or run:
  ds init --force
```

## 13. Implementation Preferences

Preferred implementation language: choose based on repository context.

If starting from scratch, Go is preferred for a portable CLI.

Recommended Go libraries:

* CLI: `spf13/cobra` or `urfave/cli`
* SQLite: `modernc.org/sqlite` or `mattn/go-sqlite3`
* YAML: `gopkg.in/yaml.v3`
* Markdown/frontmatter parsing: simple custom parser is acceptable
* ULID: `github.com/oklog/ulid/v2` or equivalent

Avoid overengineering.

## 15. Acceptance Criteria

### Initialization

* `ds init` creates `~/.devspecs/devspecs.db`.
* `ds init` creates `.devspecs/config.yaml` in the current repo/project.
* Re-running `ds init` does not destroy existing data.

### Scanning

* `ds scan` detects OpenSpec change proposals.
* `ds scan` detects ADR markdown files.
* `ds scan` detects generic markdown plans/specs from configured paths.
* `ds scan` creates artifacts with stable IDs.
* Re-running `ds scan` on unchanged files does not create duplicates.
* Changing a file and rescanning creates a new revision for the same artifact.

### Listing/Showing

* `ds list` shows indexed artifacts.
* `ds show <id>` displays useful metadata and content.
* `ds show <id> --json` emits valid JSON.

### Finding/Resolving

* `ds find <query>` returns matching artifacts by title/path/body.
* `ds resolve <id>` returns canonical source information.
* Partial IDs may be supported if unambiguous, but full IDs must always work.

### Context Export

* `ds context <id>` outputs agent-ready markdown.
* Context includes ID, title, kind, status, source, extracted criteria/tasks if available, and source content.

### Capture

* `ds capture <path>` creates or updates an artifact for a specific file.
* `ds capture <path> --kind plan` respects the provided kind.

### Status/Links

* `ds status <id> approved` updates artifact status.
* `ds link <id> <target>` stores link metadata.
* `ds show <id>` displays links.

### JSON

* Major commands support `--json`.
* JSON output should be stable enough for future automation.

## 16. Example End-to-End Flow

Given a repo with:

```txt
openspec/changes/add-sso/proposal.md
openspec/changes/add-sso/tasks.md
docs/adrs/0001-use-authjs.md
plans/refactor-auth.md
```

User runs:

```bash
ds init
ds scan
ds list
```

Expected:

```txt
ID            KIND              STATUS      TITLE
ds_01JY...    openspec_change   proposed    Add SSO
ds_01JY...    adr               accepted    Use Auth.js
ds_01JY...    plan              unknown     Refactor auth
```

User runs:

```bash
ds context ds_01JY...
```

Expected:

* Agent-ready markdown context.
* Includes source content.
* Includes extracted tasks from OpenSpec tasks file if available.

User runs:

```bash
ds link ds_01JY... https://github.com/acme/backend/pull/42
ds status ds_01JY... implementing
ds show ds_01JY...
```

Expected:

* Status changed.
* GitHub PR link visible.

## 17. Future Compatibility Notes

Design v0 so these future features are easy:

### Cloud Sync

Artifacts should already have globally unique IDs.

* artifacts
* revisions
* sources
* links
* statuses

### MCP

JSON output and `ds context` should make MCP tools straightforward later.

Future MCP tools:

```txt
devspecs_search
devspecs_get
devspecs_context
devspecs_link
devspecs_update_status
```

### GitHub/Linear

Links table should support arbitrary external targets.

Future integrations can validate and enrich links.

### Kleio

DevSpecs artifacts should become explicit intent anchors in Kleio.

Future Kleio ingestion should consume:

* artifact created
* revision added
* status changed
* link added
* implementation claim added

Do not implement these events yet, but keep schema clean enough to support them.

## 21. Definition of Done

v0 is done when a developer can:

1. Install/run the CLI locally.
2. Initialize DevSpecs in an existing repo.
3. Scan existing OpenSpec/ADR/markdown planning artifacts.
4. See a list of detected artifacts.
5. Resolve any artifact by stable ID.
6. Export agent-ready context for an artifact.
7. Capture a one-off markdown plan.
8. Mark status manually.
9. Link an artifact to an external URL.
10. Re-scan without creating duplicates.
11. Change a source file and see a new revision tracked.

## Explicit Todo Extraction

DevSpecs v0 should extract markdown checklist items from supported artifacts and store them as artifact todos.

Todos are source-derived observations, not independent project-management tasks.

The source file remains authoritative in v0.

Supported syntax:

- [ ] Incomplete task
- [x] Completed task
- [X] Completed task

Commands:

- ds todos
- ds todos <artifact-id>

Todos should be included in:

- ds show <id>
- ds context <id>
- JSON output where relevant

Do not implement owners, due dates, dependencies, comments, assignment, GitHub/Linear sync, or approval workflow in v0.

The boundary should be:

v0 DevSpecs todo model = extracted checklist observability
not task management
not workflow state
not Linear replacement
