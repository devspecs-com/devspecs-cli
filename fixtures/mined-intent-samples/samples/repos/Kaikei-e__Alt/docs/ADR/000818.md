# ADR-000818: recap-evaluator を CPU-only PyTorch + 2-stage uv sync で再構築しイメージを 8.07 GB から 1.42 GB に圧縮する

## Status

Accepted

## Date

2026-04-20

## Affected Services

- recap-evaluator (Python 3.14 / FastAPI / uvicorn — evaluation worker, CPU only)
- .github/workflows/docker-build.yaml (CI image-size / smoke-check regression gate)

## Context

`ghcr.io/kaikei-e/alt-recap-evaluator:sha-*` のイメージが **8.07 GB** に膨れ上がっており、Alt モノレポ内で最大の container image になっていた。`compose/recap.yaml` では recap-evaluator に GPU reservation は無く (recap-subworker だけが nvidia driver を予約)、サービス自体は CPU でしか評価演算を行わない設計であるにもかかわらず、CUDA wheel が約 5 GB 積まれていた。

原因は `recap-evaluator/pyproject.toml` の `bert-score>=0.3.13` が transitive dependency として `torch` を引くときに default PyPI index を参照し、`torch==2.10.0` が `nvidia-cublas-cu12` / `nvidia-cudnn-cu12` / `nvidia-nccl-cu12` など計 16 の CUDA wheel を依存として引き込んでいたことにある。`uv.lock` 上で `nvidia-*` パッケージが 36 件、総パッケージが 127 に達していた。

先行する [[000812]] で news-creator が同じ問題を解決済みで、CPU-only PyTorch を `[[tool.uv.index]]` + `[tool.uv.sources]` で pin する pattern を確立していた。[[000813]] は CI に `Enforce image-size budget and smoke-check core imports` step を拡張し、news-creator を 1800 MiB で gate している。recap-evaluator は同じ戦略を適用可能だが、以下の 2 点で news-creator と条件が異なるため個別の検証が必要だった:

1. **torch が transitive-only である**。news-creator は `sentence-transformers` だけでなく `torch>=2.11.0` も direct dependency として宣言していた。一方 recap-evaluator は `bert-score` 経由でしか torch を要求していないため、`[tool.uv.sources]` の pin が解決時に適用されない（uv の sources は「プロジェクトが直接宣言した requirement」にのみ作用する）。最初の `uv lock` 試行で torch が `https://pypi.org/simple` に resolve され続けて CUDA wheel が全く減らなかったことで確認できた。
2. **evaluation-only service でありリアル推論パスは無い**。bert-score の内部 model (`cl-tohoku/bert-base-japanese-v3`, `microsoft/deberta-xlarge-mnli`) は HF Hub から runtime で `HF_HOME` に lazy download される。イメージ内に model weight を焼き込む必要は無く、CPU-only wheel でサイズだけ削ればよい。

また `pyproject.toml` の `pandas` / `openpyxl` は Report Generation の将来用として宣言されていたが、`src/recap_evaluator/` 配下のどのファイルも import しておらず、transitive 依存として bert-score 経由で pandas が再導入される構造になっていた。dead direct dependency を剥がしても挙動は変わらない。

## Decision

以下を採用する:

### 1. torch の CPU-only wheel を `[tool.uv.sources]` で pin する

`recap-evaluator/pyproject.toml` に以下を追加する:

```toml
requires-python = ">=3.14,<3.15"

dependencies = [
    # ...既存...
    "torch>=2.11.0",   # direct に宣言し sources pin を有効化する
    "bert-score>=0.3.13",
    # ...
]

[[tool.uv.index]]
name = "pytorch-cpu"
url = "https://download.pytorch.org/whl/cpu"
explicit = true

[tool.uv.sources]
torch = [{ index = "pytorch-cpu" }]
```

併せて direct dependency から未使用の `pandas` / `openpyxl` を削除する。`uv lock` を再実行すると 127 → 109 packages に縮み、`nvidia-*` と `triton` が消え、torch の source が `https://download.pytorch.org/whl/cpu` に切り替わる。

### 2. Dockerfile を 2-stage `uv sync` に書き換える

`recap-evaluator/Dockerfile` を `# syntax=docker/dockerfile:1.7` 宣言付きの 2 段階ビルドに書き換える。builder stage では `build-essential` を仮置きして wheel の再ビルドに備え、`--mount=type=cache,id=uv-recap-evaluator,target=/root/.cache/uv` で uv cache を named cache mount に切り出す（他サービスと cache を混ぜないため `id` を service 専用に指定する）。

layer cache の再利用を最大化するため 2 段階 sync を採用する:

- 1 段目: `pyproject.toml` + `uv.lock` だけ COPY してから `uv sync --frozen --no-dev --no-install-project` で依存関係だけ入れる。
- 2 段目: `src/` を COPY してから `uv sync --frozen --no-dev --no-editable` を再実行してプロジェクト本体を `.venv` に焼く。

builder stage の末尾で `numpy/scipy/sklearn/transformers/torch/bert_score/rouge_score/pandas` の各 `site-packages/<pkg>/tests/` と `torch/include/`（C++ ヘッダ、約 120 MiB）と `*.pyi`（type stub、runtime 不要）を削除する。`__pycache__` と `*.dist-info/METADATA` は削らない（前者は `UV_COMPILE_BYTECODE=1` の成果物、後者は `transformers` の `importlib.metadata` で必要）。

runtime stage は `python:3.14-slim` に `ca-certificates` + `curl`（healthcheck 用）のみ追加し、非 root の `evaluator` user を作成し、`/app/.venv` と `src/` だけ COPY する。`OMP_NUM_THREADS=1` / `MKL_NUM_THREADS=1` で BLAS の thread 展開を抑制し、評価 job の並列度は uvicorn の worker 管理側に委ねる。

### 3. CI image-size gate と smoke-check を拡張する

`.github/workflows/docker-build.yaml` の `Build and push` step の `load:` 条件と `Enforce image-size budget and smoke-check core imports` step の `if:` 条件を `["recap-subworker","news-creator","recap-evaluator"]` に拡張する。`MAX_MB` に `recap-evaluator) MAX_MB=2500` を追加する。実測 1353 MiB だが、bert-score が runtime で追加 wheel を resolve する余地と、将来の transformers / scikit-learn 更新による緩やかな膨張を見越して +1000 MiB 強のヘッドルームを取る。

smoke-check は `elif [ "$SERVICE" = "recap-evaluator" ]; then` 分岐を追加し、`import asyncpg, torch, bert_score, rouge_score, recap_evaluator` を実行する。`import recap_evaluator.main` は `Settings()` 検証で `RECAP_DB_DSN` を要求するためスキップし、トップレベル package の import だけで wheel 層の健全性を確認する。`torch.cuda.is_available() == False` と `nvidia` / `triton` modulesearch 不在を assert する点は [[000813]] と同じ。

### 検討した代替案と却下理由

- **bert-score を ONNX-based MiniLM で再実装する**: 約 700 MiB まで縮む可能性があるが、BERTScore の P/R/F1 計算ロジックを ONNX embedding 上で再実装する必要があり（約 80 LOC）、評価結果の semantic contract が変わる。本 ADR の目的は image slim であり評価アルゴリズムの置換ではないため却下。budget 2500 MiB で当面十分だが、運用で天井に当たる兆候が出たら ADR を分けて検討する。
- **bert-score / transformers 自体を runtime optional にして `bert_score` を extras に逃がす**: 現状 `src/recap_evaluator/evaluator/bertscore_eval.py` は既に `try/except ImportError` で guard されているが、`config.py` にも weight 0.25 で固定されており、extras 化すると config/weight/scheduler の同期が増えて ops が複雑になる。image 側で解決可能な以上、config レイヤに副作用を広げない。
- **HF model weight を pre-download して image に焼き込む**: 起動時の model fetch による initial latency を消せるが、image が +1.5 GB 程度膨らむ上、model 更新のたびに re-build が必要になる。`HF_HOME` を persistent volume にした現構成で十分。

## Consequences

### Pros

- recap-evaluator の image が 8.07 GB → 1.42 GB (約 -83%) に縮む。GHCR の push / pull 帯域と Kaikei-e/alt-deploy の rollout 時間が同比率で短縮される。
- `.venv` だけを runtime に渡す 2-stage build により builder stage の `build-essential` / uv cache / 中間 wheel が runtime image に漏れない。attack surface が縮み OWASP A06 (vulnerable components) の露出が減る。
- `UV_COMPILE_BYTECODE=1` でビルド時に `.pyc` を前焼きしつつ `UV_LINK_MODE=copy` で cache mount と layer の FS を分離しているため、container 初回起動時の import latency も改善する。
- CI gate により今後 nvidia wheel が再混入しても smoke-check で `assert not torch.cuda.is_available()` と `nvidia` import 不在が検出され、`MAX_MB=2500` 超過は `::error::` で fail する。先行する [[000813]] と同じ境界を recap-evaluator にも引けた。
- 未使用の `pandas` / `openpyxl` の direct declaration を剥がし、Report Generation の実装開始時には明示的に再導入する整理ができた。

### Cons / Tradeoffs

- `torch` を direct dependency として宣言する必要がある。本来 `bert-score` が pull していたものだが、uv の `[tool.uv.sources]` が transitive request に作用しない仕様のため、workspace root に torch を持たせないと CPU wheel への routing が効かない。pyproject の artistic integrity を若干損なうが、コメントで理由を明示して運用する。
- bert-score の多言語 model（日本語 `cl-tohoku/bert-base-japanese-v3`、英語 `microsoft/deberta-xlarge-mnli`）は引き続き runtime に HF Hub から download される。初回起動時に `HF_HOME` が空なら 1.5 GB 程度の fetch が発生するため、compose 側で `HF_HOME` を named volume として永続化しているか確認が要る（現 `compose/recap.yaml` は `mem_limit: 2g` だけで明示的な volume は無いため、運用時は `HF_HOME` を vol に bind する follow-up が望ましい）。
- `OMP_NUM_THREADS=1` を強制したため、評価 job 内で numpy / scipy が BLAS 並列を活かせない。evaluation は 1 件ずつ完結する I/O bound 中心のため影響は小さいと判断したが、評価量が上振れして CPU bound に寄ったときは container 再設計が必要になる。
- CI budget 2500 MiB は実測 1353 MiB から +1147 MiB の余裕を持たせた値だが、transformers の major upgrade や scikit-learn の BLAS backend 変更で急に膨張するリスクは残る。`docker-build.yaml` の gate で検知はできるが、budget tuning を四半期単位で見直す運用コストは今後増える。
- uv `[tool.uv.sources]` は 0.5 系の resolver 挙動に依存する。uv を 0.11.x から更新する際は `uv lock --dry-run` で torch の `source` が `https://download.pytorch.org/whl/cpu` のままであることを CI の先で確認する必要がある。
